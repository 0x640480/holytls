# Cross-client TLS-impersonation HTTP benchmark — results

> Generated by `bench/scripts/run_benchmark.sh` (reduced sample: RUNS=2, 1200/400 reqs).
> Numbers are directional (WSL2); the value here is the method + the findings.

Clients: **holytls** (C), **tls-client** (Go), **httpcloak** (Go), **wreq** (Rust). `—` = not run (e.g. H3 for tls-client/wreq, which lack QUIC). `⚠` = a run negotiated a different protocol than requested.

## Key observations (v1 sample on WSL2 — directional; re-run on bare metal to publish)

- **Steady-state (single-thread = the fair comparison) is competitive.** On H2 at
  C=8–64 all four cluster in a similar band (~17–37k req/s for 1K bodies); holytls
  leads H2 at C=8 and is on par at C=64. H1 varies more by client.
- **holytls H3 collapses at ≥32 concurrent QUIC streams** — C=16: ~35k req/s, but
  C≥32: ~27 req/s with p99 ≈ 30 s (errors=0; the tail just *stalls* ~30 s). H2
  scales fine (31k at C=64), so this is QUIC-specific — a concurrency stall (likely
  MAX_STREAMS / flow-control) the benchmark surfaced. **A real holytls issue to
  investigate, not a harness artifact.**
- **wreq H1 cold = 22 req/s** — wreq's fresh-connection-per-request H1 path is
  anomalously slow (its H2 cold is normal ~1.2k); worth a look.
- **httpcloak cold is fast because it 1-RTT resumes** (keeps TLS tickets) while the
  others full-handshake — cold is "new-connection cost", not a handshake race.
- **Full-power** shows Go/Rust scaling across cores (httpcloak 62k, wreq H1 31k)
  while holytls stays a single loop by design — not a head-to-head (see caveats).

## Caveats (read before quoting any number)

- **Single-loop vs multi-threaded.** holytls is one libuv loop; tls-client/httpcloak (Go) and wreq (Rust) are multi-threaded runtimes. The single-thread group is the fair comparison; full-power is per-runtime and not head-to-head.
- **Different TLS/H2/H3 stacks.** holytls = lexiforest/BoringSSL + nghttp2 + ngtcp2; wreq = upstream BoringSSL; tls-client = Go utls + net/http2; httpcloak = sardanioss utls + quic-go forks. This benchmarks end-to-end client performance, not just TLS.
- **Cold resumption** (above): httpcloak resumes; others full-handshake. Keepalive is the clean steady-state comparison.
- **HTTP/3** is holytls vs httpcloak only (tls-client/wreq have no QUIC — a feature gap, not a loss).
- **Environment.** See above; on WSL2 the CPU governor/turbo are uncontrollable and timers are virtualized — treat WSL2 numbers as directional and re-run on bare metal for publication.
- **wreq** trusts the origin via skip-verify (no custom-CA API); the full handshake still runs, only the microsecond chain check is skipped.
