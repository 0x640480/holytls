# Cross-client TLS-impersonation HTTP benchmark — results

> Generated by `bench/scripts/run_benchmark.sh` (default config: RUNS=3, 2000/800 reqs).
> Numbers are directional (WSL2); the value here is the method + the findings.
> Re-run on bare metal before quoting any absolute number.

Clients: **holytls** (C), **tls-client** (Go), **httpcloak** (Go), **wreq** (Rust). `—` = not run (e.g. H3 for tls-client/wreq, which lack QUIC). `⚠` = a run negotiated a different protocol than requested.

## Key observations (single-thread = the fair comparison)

- **HTTP/3 leads, by a lot.** holytls H3 keepalive (1K, C=64) is **47.8k req/s vs
  httpcloak 16.7k** (~2.9×), and ~1.85× at C=8 — the only other H3 client. (An
  earlier H3 concurrency stall at ≥32 streams was root-caused and fixed: pooled H3
  over-capacity submits now park instead of leaking inflight.)
- **HTTP/2 steady-state is strong.** H2 1K C=64: holytls **31.3k** — 2nd of four
  (tls-client 35.8k, wreq 30.7k, httpcloak 16.6k), and it scales cleanly with
  concurrency (3.3k→19.3k→31.3k for C=1→8→64).
- **Leanest memory by a wide margin.** holytls RSS is ~9–12 MB across most cells vs
  18–83 MB for the Go clients and up to ~26 MB for wreq — the arena model paying off.
- **Lowest tail latency at low concurrency** (H3 p99 0.49 ms @ C=1, H2 0.44 ms) and
  the most run-to-run consistency (tight min/max vs tls-client/wreq swings).
- **Weak spot 1 — concurrent H1 doesn't scale.** H1 keepalive is flat ~5.4k req/s
  at C=1/8/64: holytls serializes on a single connection (no H1 multiplexing, and
  this config caps conns/origin at 1), while the others fan out across connections
  (wreq 69.7k at C=64). Concurrent H1 needs multi-connection pooling.
- **Weak spot 2 — large-body (100K) per-connection throughput is ~2× behind.** On a
  single H2 connection holytls does ~3.3k req/s (~330 MB/s) vs httpcloak ~6.1k
  (~610 MB/s); the ~3.3k ceiling is protocol-independent (h1/h2/h3 all land there),
  pointing at the response-body receive/copy path, not the protocol.
- **Weak spot 3 — cold/new-connection is ~3× behind httpcloak** (which 1-RTT resumes;
  holytls full-handshakes by default though it *supports* resumption/0-RTT). Connection
  setup is a cost center for many-origin / non-keepalive workloads.
- **Full-power is not a head-to-head** — holytls is one libuv loop by design; the Go/Rust
  runtimes spread across cores there. See caveats.

## Caveats (read before quoting any number)

- **Single-loop vs multi-threaded.** holytls is one libuv loop; tls-client/httpcloak (Go) and wreq (Rust) are multi-threaded runtimes. The single-thread group is the fair comparison; full-power is per-runtime and not head-to-head.
- **Different TLS/H2/H3 stacks.** holytls = lexiforest/BoringSSL + nghttp2 + ngtcp2; wreq = upstream BoringSSL; tls-client = Go utls + net/http2; httpcloak = sardanioss utls + quic-go forks. This benchmarks end-to-end client performance, not just TLS.
- **Cold resumption** (above): httpcloak resumes; others full-handshake. Keepalive is the clean steady-state comparison.
- **HTTP/3** is holytls vs httpcloak only (tls-client/wreq have no QUIC — a feature gap, not a loss).
- **Environment.** See above; on WSL2 the CPU governor/turbo are uncontrollable and timers are virtualized — treat WSL2 numbers as directional and re-run on bare metal for publication.
- **wreq** trusts the origin via skip-verify (no custom-CA API); the full handshake still runs, only the microsecond chain check is skipped.

Full data lives under `bench/results/<timestamp>/` (`aggregate.csv` + `json/`); this file is one representative run committed for reference.
